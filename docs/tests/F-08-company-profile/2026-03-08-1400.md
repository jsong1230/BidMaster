## 보안

### Critical

**[SEC-001] 인-메모리 저장소를 프로덕션 런타임에서 사용**  
파일: `backend/src/services/company_service.py` 라인 19-24  
`_companies`, `_company_members`, `_performances`, `_certifications`, `_users`가 모듈 수준 전역 딕셔너리로 선언되어 있으며, `CompanyService(db=None)`으로 인스턴스화된다 (`companies.py` 라인 35). 이 구조는 프로세스 재시작 시 모든 데이터가 소실되고, 다중 워커 환경(Gunicorn, uvicorn workers)에서 워커 간 데이터가 공유되지 않아 인가 우회가 발생할 수 있다. 예: 워커 A에서 회사 등록 후 워커 B로 요청이 라우팅되면 `_companies`에 해당 회사가 없어 404가 반환된다.

**[SEC-002] API 레이어에서 인-메모리 내부 상태를 직접 임포트·조작**  
파일: `backend/src/api/v1/companies.py` 라인 153-163  
`get_my_company` 엔드포인트가 `from src.services.company_service import _users, CompanyResponse`를 호출하여 서비스의 내부 저장소에 직접 mock 객체를 삽입한다. 이는 서비스 레이어 캡슐화를 파괴하고, 런타임에서 외부 요청이 `_users` 딕셔너리를 오염시킬 수 있다.

**[SEC-003] 테스트 전용 파라미터가 프로덕션 API에 노출**  
파일: `backend/src/services/company_service.py` 라인 152, 366, 573  
`_force_duplicate`, `_force_max_representative`, `_force_existing_member`, `_force_other_company` 등 테스트 제어용 키/파라미터가 실제 서비스 메서드에 남아 있다. 클라이언트가 요청 본문에 `{"_force_duplicate": true}`를 전송하면 강제로 409 응답을 유발할 수 있다. 서비스 레이어에서 사용자 입력 딕셔너리를 그대로 받아 `.get("_force_duplicate")`를 검사하므로, 악의적 사용자가 특정 에러 분기를 강제 실행할 수 있다.

**[SEC-004] page/pageSize 파라미터에 입력 검증 없음**  
파일: `backend/src/api/v1/companies.py` 라인 300-301, 525, 705  
`page = int(params.get("page", 1))`, `page_size = int(params.get("pageSize", 20))`에 예외 처리나 범위 검증이 없다. `?page=abc` 전송 시 `ValueError`가 처리되지 않아 500 응답이 반환된다. `page_size=99999` 전송 시 인-메모리 전체 순회가 발생한다. (현재 인-메모리 구현에서는 성능 영향이 제한적이나, DB로 마이그레이션 시 즉각적인 보안/성능 이슈가 된다.)

### Warning

**[SEC-005] update_performance에서 날짜 변경 시 교차 검증 없음**  
파일: `backend/src/services/company_service.py` 라인 481-525  
`update_performance`는 `startDate`와 `endDate` 중 하나만 변경하는 경우 기존 값과의 교차 검증을 수행하지 않는다. `startDate`만 변경하여 `endDate`보다 나중 날짜로 설정해도 통과된다.

**[SEC-006] update_certification에서 날짜 교차 검증 없음**  
파일: `backend/src/services/company_service.py` 라인 729-772  
`update_certification`은 `issuedDate`와 `expiryDate`를 각각 독립적으로 업데이트하며, 수정 후의 날짜 순서(issued <= expiry)를 검증하지 않는다.

**[SEC-007] update_company에서 scale 값 검증 없음**  
파일: `backend/src/services/company_service.py` 라인 339  
`create_company`에서는 `scale`이 `("small", "medium", "large")` 범위인지 검사하나, `update_company`에서는 해당 검증이 없다. `{"scale": "enterprise"}` 전송 시 그대로 저장된다.

**[SEC-008] documentUrl에 URL 형식 검증 없음 (백엔드)**  
파일: `backend/src/services/company_service.py` 라인 423, 668  
`documentUrl`이 임의 문자열로 저장 가능하다. 프론트엔드 `type="url"` 속성은 클라이언트 측 검증이므로 우회 가능하다.

### 통과

- JWT Bearer 토큰 인증이 모든 엔드포인트에 적용되어 있음 (`get_current_user_payload`)
- RBAC: owner/admin/member 역할 분리 구현
- 사업자등록번호 체크섬 검증 구현 (`validate_business_number`)
- 회사 소속 확인 후 데이터 접근 구조 (소속 회사 데이터만 접근 가능)
- 프론트엔드 입력 필드에 XSS 위험이 있는 `dangerouslySetInnerHTML` 미사용

## 성능

### Critical

**[PERF-001] 인-메모리 전체 순회로 집계 (확장 불가 구조)**  
파일: `backend/src/services/company_service.py` 라인 262-275  
`get_my_company`에서 `member_count`, `performance_count`, `certification_count`를 계산할 때 전체 딕셔너리를 순회한다. 현재 인-메모리 구조에서는 문제가 없으나, DB 마이그레이션 시 전체 테이블 스캔 또는 비효율적 쿼리로 이어질 수 있는 패턴이다. `list_performances`, `list_certifications`, `list_members`에서도 동일하게 전체 딕셔너리를 순회한 후 슬라이싱한다.

### Warning

**[PERF-002] CertificationTab에서 TODAY를 모듈 상수로 고정**  
파일: `frontend/src/components/company/CertificationTab.tsx` 라인 20  
`const TODAY = new Date()`가 모듈 최초 임포트 시점에 한 번만 평가된다. 사용자가 페이지를 오래 열어 두면 만료 판단 기준 날짜가 갱신되지 않는다. 컴포넌트 마운트 시점 또는 렌더 시점에 `new Date()`를 호출해야 한다.

**[PERF-003] PerformanceTab에서 `filterStatus`를 클라이언트 측에서 필터링**  
파일: `frontend/src/components/company/PerformanceTab.tsx` 라인 511-514  
실적 목록 전체를 API에서 받아 클라이언트에서 필터링한다. 데이터가 많아지면 불필요한 전송량이 증가한다. (현재는 `pageSize` 기본값 20이라 영향이 제한적이나, 필터가 서버 측에 전달되는 `listPerformances` API가 이미 존재함에도 활용하지 않는다.)

**[PERF-004] MemberTab에서 `fetchMembers` 의존성 배열에 함수 참조 포함**  
파일: `frontend/src/components/company/MemberTab.tsx` 라인 144-146  
`useEffect`의 의존성 배열에 `fetchMembers`가 포함되어 있으나, Zustand 스토어의 함수는 매 렌더마다 새로 생성되지 않아 실질적 문제는 없다. 동일한 패턴이 `PerformanceTab` 라인 507-509, `CertificationTab` 라인 330-332에도 존재한다.

### 통과

- 목록 API 페이지네이션 구현 (page, pageSize)
- 대표 실적 낙관적 업데이트(optimistic update) + 실패 시 롤백 구현
- Skeleton 로딩 UI 구현 (CompanyBasicInfoTab, MemberTab, PerformanceTab, CertificationTab)

## 설계/문서 일치

### 설계 불일치

**[DESIGN-001] DB 없이 인-메모리 저장소로 구현 (DB 설계 미반영)**  
design.md 2절에서 users.company_id FK, companies 테이블, company_members, performances, certifications 테이블이 명시되어 있고, 3절에서 Alembic 마이그레이션 순서까지 정의되었으나, 실제 구현은 SQLAlchemy 모델/스키마 없이 전역 딕셔너리로 대체되었다. `backend/src/models/`에 company.py가 존재하지 않는다.

**[DESIGN-002] CompanyService가 AppException을 자체 정의 (공통 예외 클래스 미사용)**  
파일: `backend/src/services/company_service.py` 라인 10-16  
설계서(F-07 design.md 패턴)에서 `AppException`은 `src/core/` 모듈에서 공통으로 제공되어야 한다. 그러나 `company_service.py`가 자체적으로 `AppException`을 재정의하고 있어, 다른 서비스의 `AppException`과 별개의 클래스가 된다. 예외 핸들러에서 공통으로 처리할 수 없는 구조다.

**[DESIGN-003] Pydantic 스키마 없이 `dict[str, Any]` 사용**  
design.md 5절 API 설계에서 요청/응답 스키마를 명시하였으나, `backend/src/schemas/company.py`가 존재하지 않는다. 서비스 메서드가 모두 `data: dict[str, Any]`를 받아 `.get()`으로 필드를 추출한다. 이로 인해 입력 타입 검증, 직렬화/역직렬화, OpenAPI 문서 자동 생성이 모두 불가능하다.

**[DESIGN-004] `get_company_service()`가 `db=None`을 전달**  
파일: `backend/src/api/v1/companies.py` 라인 34-35  
설계서에서 DB 의존성 주입(SQLAlchemy async session)이 명시되어 있으나, 실제 구현은 `CompanyService(db=None)`을 반환한다. `create_company`에서 DB 조회를 시도할 때 `self.db`가 `None`이면 예외가 발생하고 `except Exception`으로 묵살된다 (라인 190-193).

### 문서 불일치

**[DOC-001] 인수조건 "회사 프로필 저장 시 pgvector 임베딩 저장" 미구현**

**[DOC-002] MemberTab에서 멤버 제거 기능 UI만 존재, API 미연결**  
파일: `frontend/src/components/company/MemberTab.tsx` 라인 233-239  
"제거" 버튼이 렌더되나 `onClick` 핸들러가 없다. `member-store.ts`에도 `removeMember` 액션이 없고, `company-api.ts`에도 멤버 제거 API 함수가 없다. features.md에 멤버 제거 인수조건은 없으나, UI 버튼이 비활성 상태로 노출되면 사용자 혼동을 유발한다.
