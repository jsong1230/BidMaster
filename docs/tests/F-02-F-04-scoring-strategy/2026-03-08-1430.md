## 보안

### Critical

**[SEC-01] 프로덕션 코드에서 `unittest.mock.AsyncMock` 사용**

- 파일: `backend/src/api/v1/bids.py` (484, 486, 545, 547행)
- 내용: `GET /bids/{id}/strategy` 및 `POST /bids/{id}/strategy/simulate` 엔드포인트에서 `from unittest.mock import AsyncMock`을 import 후 `BiddingStrategyService(db=AsyncMock())`으로 서비스를 초기화하고 있음. `unittest.mock`은 테스트 전용 라이브러리로 프로덕션 코드에 포함되면 안 됨. 의도치 않은 DB 호출 무시, 실제 오류 은폐 가능성.

```python
# 484~487행 (strategy 엔드포인트)
from src.services.bidding_strategy_service import BiddingStrategyService
from unittest.mock import AsyncMock

service = BiddingStrategyService(db=AsyncMock())
```

**수정 방법**: `db=None` 또는 실제 DB 세션을 주입. `AsyncMock`을 제거하고 `BiddingStrategyService(db=None)`으로 교체 (서비스 내부 코드가 DB 없이도 인메모리 데이터로 동작함을 확인 후 적용).

---

**[SEC-02] 공고 ID 검증이 인증 검사보다 먼저 수행됨 (정보 노출)**

- 파일: `backend/src/api/v1/bids.py` (378~396행, get_bid 핸들러)
- 내용: `GET /bids/{bid_id}` 에서 UUID 형식 검사가 먼저 수행되고 이후 JWT 인증이 수행되어, 인증 없이도 UUID 형식인지 아닌지 구분 가능. 공격자가 인증 없이 입력값 유효성 피드백을 받을 수 있음. scoring, strategy 엔드포인트도 동일 패턴.
- 심각도: 낮으나 패턴 일관성 문제 (인증 먼저, 검증 이후가 표준).

**수정 방법**: 인증 검사(`_get_current_user`)를 UUID 형식 검사 앞으로 이동.

### Warning

**[SEC-03] 인메모리 캐시 `_SCORING_CACHE`에 TTL(만료 시간) 없음**

- 파일: `backend/src/services/scoring_service.py` (54, 115~116, 199행)
- 내용: 스코어링 결과를 `{user_id}:{bid_id}` 키로 딕셔너리에 영구 저장. 서버 재시작 전까지 캐시가 만료되지 않으므로, 공고 상태나 회사 정보가 변경되어도 오래된 결과를 반환할 수 있음. 메모리 누수 가능성도 있음.

**수정 방법**: 캐시 엔트리에 `analyzed_at` 기반 TTL(예: 1시간)을 적용하거나 Redis로 교체.

---

**[SEC-04] SQL Injection 잠재 위험 (raw SQL + f-string)**

- 파일: `backend/src/services/scoring_service.py` (634~644행, `_get_win_history` DB 폴백)
- 내용:
```python
where_clause = " OR ".join(conditions)
rows = await self.db.execute(
    text(f"""
        SELECT wh.*, b.organization AS _organization, ...
        WHERE {where_clause}
        ...
    """),
    params,
)
```
`conditions` 리스트 자체는 하드코딩된 문자열이지만, `where_clause`를 f-string으로 직접 SQL에 삽입하는 패턴. `conditions`가 외부 입력 기반으로 동적 확장될 경우 injection 위험. 현재는 조건이 고정값(`:org`, `:cat`)이나 패턴 자체가 위험.

**수정 방법**: f-string 대신 SQLAlchemy ORM 또는 `text()` + 고정 조건 분리 방식으로 교체.

---

**[SEC-05] 스코어링 엔드포인트에 구독 여부/RBAC 미검증**

- 파일: `backend/src/api/v1/bids.py` (596~654행)
- 내용: 스코어링 및 투찰 전략 API가 JWT 인증은 하지만 역할(role) 또는 구독 여부 검사가 없음. features.md 기준 이들은 핵심 유료 기능(F-09 결제 의존). 현재는 모든 인증된 사용자가 제한 없이 접근 가능.
- 현재 단계(개발 중)에서는 허용 가능하나 프로덕션 전에 반드시 처리 필요.

---

### 통과

- JWT 검증: `decode_token`이 만료(`ExpiredSignatureError`), 서명 변조(`JWTError`) 모두 처리
- Bearer 토큰 파싱: `Authorization: Bearer` 접두사 검증 정상
- 비밀번호 해싱: bcrypt 적용, 평문 저장 없음
- 하드코딩된 시크릿: `secret_key`를 `settings`(환경변수)에서 로드, 코드에 하드코딩 없음
- XSS: 프론트엔드에서 `dangerouslySetInnerHTML` 미사용
- CORS/SameSite: 별도 CORS 미들웨어 구성(별도 파일 확인 대상이나 코드 내 직접 노출 없음)

**[QUAL-05] `SimulationResult.riskLevel` 타입 불일치**

- 백엔드 `SimulationResult` 스키마 (strategy.py 69행): `riskLevel: str` (미제한)
- 프론트엔드 타입 (strategy.ts 67행): `riskLevel: 'safe' | 'moderate' | 'aggressive' | 'over_safe' | 'extreme' | 'unknown'`
- 백엔드에서 `_determine_risk_level`이 반환하는 값: `"over_safe"`, `"extreme"`, `"moderate"` 등 — `"unknown"`은 백엔드에서 반환되지 않음. 반면 백엔드 Pydantic 스키마에는 `str`로만 정의.

**수정 방법**: 백엔드 `SimulationResult.riskLevel`을 `Literal["safe", "moderate", "aggressive", "over_safe", "extreme"]`로 타입을 구체화.
