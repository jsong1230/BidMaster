## 보안

### Critical

**[C-SEC-01] 테스트 토큰 하드코딩 (bids.py 28~44행)**
- `_TEST_TOKENS` 딕셔너리에 `test-token-owner`, `test-token-member` 등 예측 가능한 고정 토큰이 프로덕션 코드에 존재함
- 누구든 해당 값을 알면 인증 없이 owner/member 권한으로 모든 API에 접근 가능
- 대상 파일: `backend/src/api/v1/bids.py:28-44`

**[C-SEC-02] 인메모리 전역 변수로 토큰 검증 우회 가능 (bids.py 131행)**
- `_get_current_user()` 함수에서 실제 JWT 검증 이전에 `_TEST_TOKENS` 딕셔너리를 먼저 확인함
- 테스트 목적의 분기가 프로덕션 코드에 포함된 구조 자체가 인증 우회 경로가 됨
- 대상 파일: `backend/src/api/v1/bids.py:131-132`

**[C-SEC-03] API 키가 HTTP 요청 파라미터로 평문 전송 (bid_collector_service.py 120행)**
- `ServiceKey` 를 URL 쿼리 파라미터로 전송하여 로그, 프록시, 브라우저 히스토리에 노출될 위험
- 설계서는 API Key 기반 인증만 언급하며 전송 방식을 명시하지 않았으나, 공공데이터포털 API 규격상 파라미터 전달은 불가피함 — 최소한 httpx 로그 레벨에서 마스킹 처리 필요
- 대상 파일: `backend/src/services/bid_collector_service.py:119-124`

### Warning

**[W-SEC-01] 외부 파일 URL 검증 없이 다운로드 (bid_parser_service.py 73행)**
- `_download_file(url)` 에서 URL 유효성 검사 없이 임의 URL로 HTTP 요청을 수행함
- 악성 공고 데이터가 삽입되면 내부 네트워크(SSRF) 요청 가능
- 예: `file_url: "http://169.254.169.254/latest/meta-data/"` 등 메타데이터 엔드포인트 접근

**[W-SEC-02] `except Exception: pass` 패턴으로 DB 예외 전부 무시 (bid_collector_service.py 296행, bid_match_service.py 457행)**
- DB 오류 발생 시 예외가 완전히 숨겨져 데이터 무결성 문제를 인지할 수 없음
- 특히 `_save_bid()`에서 DB 저장 실패 시 인메모리 폴백으로 실행되어 재시작 후 데이터 유실

**[W-SEC-03] 임시 파일 미삭제 (bid_parser_service.py 80~82행)**
- `tempfile.NamedTemporaryFile(delete=False)` 로 생성된 파일이 파싱 후 삭제되지 않음
- 디스크 용량 고갈 및 민감한 공공문서 파일이 임시 디렉토리에 무기한 잔류

**[W-SEC-04] `minBudget` / `maxBudget` Decimal 변환 예외 처리 없음 (bids.py 212~213행)**
- `Decimal(min_budget_str)` 이 잘못된 값에 대해 `InvalidOperation` 예외를 발생시키면 500 응답이 반환됨
- 입력 검증이 try/except 없이 직접 수행됨

**[W-SEC-05] `sortBy` / `sortOrder` 유효성 검사 미실행 (bids.py 201~202행)**
- `VALID_SORT_BY`, `VALID_SORT_ORDER` 상수가 정의되어 있으나 실제 검증 로직이 없음
- `status`에는 검증이 적용되지만 정렬 파라미터는 무방비 상태로 `sort_key_map.get()` 폴백에만 의존

### 통과

- JWT 검증은 `decode_token()` 호출로 위임되어 있어 토큰 위조 방어 적용됨
- `bid_id` UUID 형식 검증 (`get_bid` 엔드포인트 399~409행)
- `status` 필터 화이트리스트 검증 적용
- role 기반 접근 제어 (`collect` 엔드포인트: owner 전용)

## 성능

### Critical

**[C-PERF-01] analyze_new_bids_for_all_users에서 N*M 중첩 루프 + 매 반복마다 다수의 DB 쿼리 (bid_match_service.py 119~128행)**
- 공고 수(N) x 사용자 수(M) 조합마다 `analyze_match()` 호출
- `analyze_match()` 내부에서 매 호출마다: `_get_user_company()`, `_get_bid()`, `_get_company_performances()`, `_get_company_certifications()`, `_get_bid_attachments()` — 총 5회 DB 쿼리
- 신규 공고 100건, 회사 보유 사용자 50명 기준 약 25,000 DB 쿼리 발생
- 설계서(§10.1)의 시퀀스와 일치하지만 실제 구현에서 쿼리 최적화(일괄 조회) 없음

**[C-PERF-02] `_get_bid()` 가 N*M 루프 안에서 매번 동일 공고를 반복 조회 (bid_match_service.py 363~373행)**
- 공고 1건에 대해 M명의 사용자 분석 시 동일한 bid를 M번 SELECT함
- 공고 데이터는 루프 전 한 번만 조회하면 충분

### Warning

**[W-PERF-01] TF-IDF Vectorizer가 매 분석마다 새로 생성됨 (bid_match_service.py 231행)**
- `TfidfVectorizer()` 인스턴스가 `_calculate_tfidf_similarity()` 호출마다 생성됨
- N*M 조합 수만큼 vectorizer 생성 비용 발생

**[W-PERF-02] `_is_duplicate()` 에서 인메모리 + DB 이중 확인 (bid_collector_service.py 373~397행)**
- 인메모리 딕셔너리와 DB 둘 다 조회하는 구조는 일관성을 보장하지 못함
- 수집 도중 서버 재시작 시 인메모리 저장소가 초기화되어 중복 저장 위험

**[W-PERF-03] 첨부파일 파싱 루프가 직렬 실행 (bid_parser_service.py 151~158행)**
- `parse_all_for_bid()` 에서 첨부파일을 하나씩 순차 처리
- 다운로드 + 파싱이 I/O 바운드 작업임에도 `asyncio.gather()` 미사용

**[W-PERF-04] 스케줄러가 Redis 잠금을 실제로 획득하지 않고 수집 실행 (scheduler.py 13~58행)**
- `acquire_collection_lock()` 함수가 구현되어 있으나 `scheduled_collect_bids()` 내에서 호출되지 않음
- 다중 인스턴스 배포 또는 스케줄러 재시작 시 중복 수집 발생 가능

### 통과

- `bids` 테이블 인덱스 설계서 인덱스 계획과 일치 (`idx_bids_status_deadline`, `idx_bids_bid_number` 등)
- `user_bid_matches` 복합 유니크 인덱스 및 사용자-점수 인덱스 구현됨
- API 목록 응답에 페이지네이션 적용됨 (pageSize 최대 100 제한)
- httpx timeout 30초 설정

## 설계/문서 일치

### 설계 불일치

**[D-01] 스케줄러에서 Redis 잠금 획득/해제 누락 (scheduler.py)**
- 설계서 §9.3에서 `acquire_collection_lock()` / `release_collection_lock()` 호출을 `scheduled_collect_bids()` 내 필수 구현으로 정의
- 구현에서 두 함수는 정의만 되어 있고 `scheduled_collect_bids()`에서 호출되지 않음

**[D-02] `_fetch_from_api()` 가 실제 수집 루프에서 미사용 (bid_collector_service.py)**
- 설계서 §7.1에서 `_fetch_from_api()`가 API 호출 전담 메서드로 설계됨
- 구현에서 `collect_bids()` 는 `_fetch_from_api()`를 호출하지 않고 직접 `http_client.get()`을 수행
- `_retry_with_backoff()` 도 `collect_bids()`에서 사용되지 않아 AC-05 재시도 요구사항이 실제 수집 경로에 미적용

**[D-03] `_save_bid()`의 DB 저장 실패 시 인메모리 폴백이 설계서에 없음 (bid_collector_service.py 270~318행)**
- 설계서는 DB 저장만 명시; 인메모리 폴백은 테스트 편의를 위한 구현이나 프로덕션 환경에서 데이터 유실 원인이 됨
- DB 저장 실패 시 `CollectionError`를 발생시켜야 함

**[D-04] `_upsert_match()`에서 INSERT 경로가 구현되지 않음 (bid_match_service.py 424~460행)**
- 기존 매칭 존재 시 UPDATE는 구현됨
- 신규 매칭(INSERT)은 except 블록 통과 후 `match` 객체를 그대로 반환하여 DB 미저장
- `user_bid_matches` 테이블에 실제 데이터가 기록되지 않을 수 있음

**[D-05] 수집 API(`POST /collect`)가 실제 수집을 백그라운드에서 실행하지 않음 (bids.py 475~504행)**
- 설계서 §5.5에서 "백그라운드에서 수집 실행 (응답은 즉시 반환)" 명시
- 구현에서는 202 응답만 반환하고 `BidCollectorService` 호출이 없음 (Mock 수준)

**[D-06] 설계서 §8.4의 `inqryBgnDt` / `inqryEndDt` 파라미터가 API 호출에 미포함 (bid_collector_service.py 178~184행)**
- 조회 시간 범위 없이 전체 공고를 반복 수집하면 중복 처리 오버헤드 증가

### 문서 불일치

**[DOC-01] `BidQuery` 스키마가 API 엔드포인트에서 미사용 (schemas/bid.py)**
- Pydantic 스키마로 `BidQuery`가 정의되어 있으나 `list_bids()` 엔드포인트는 `request.query_params`를 직접 파싱
- 스키마 검증 레이어가 실질적으로 동작하지 않음

**[DOC-02] `BidListItem.attachment_count` 필드명이 API 응답 키와 불일치**
- `schemas/bid.py`의 Pydantic 모델: `attachment_count` (snake_case)
- `bids.py` 실제 응답 JSON: `"attachmentCount"` (camelCase)
- 스키마와 실제 응답 형식이 다름

**[DOC-03] `BidAttachmentSchema.file_url` 필드명 불일치**
- `schemas/bid.py`: `file_url` (snake_case)
- `bids.py` 응답 및 프론트엔드 타입: `fileUrl` (camelCase)

### 인수조건 충족

- 인수조건 AC-01 (공고 DB 저장): `BidCollectorService.collect_bids()` 및 `Bid` 모델 구현됨 — 단, D-02로 인해 재시도 경로 미연결
- 인수조건 AC-02 (PDF/HWP 파싱): `BidParserService` 구현됨, graceful degradation 적용
- 인수조건 AC-03 (적합도 점수 0~100): `BidMatchService._compute_final_score()` 구현, `min(100.0, ...)` 상한 처리됨
- 인수조건 AC-04 (70점 이상 알림): `NotificationService` 스텁 연동됨, is_notified 갱신 로직 존재

### 인수조건 미충족

- 인수조건 AC-05 (지수 백오프 3회 재시도): `_retry_with_backoff()` 메서드는 구현되었으나 `collect_bids()` 의 실제 API 호출 경로에서 사용되지 않음 — **핵심 인수조건 미달**
- 인수조건 AC-01 (수집 후 관리자 알림): 3회 실패 후 `send_admin_alert()` 호출이 없음
